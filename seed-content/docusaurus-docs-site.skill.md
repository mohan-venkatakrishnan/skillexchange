# Documentation Sites that Scale with Docusaurus Skill

## 1. Philosophy

Docusaurus is easy to start and easy to wreck. The wreck always looks the same: a `sidebars.js` hand-maintained to 400 lines, three versions snapshotted in month two, a dozen swizzled components pinning you to a minor release you can never upgrade past, and a docs tree nobody can navigate because it grew by accretion.

The mental model that keeps a docs site alive: **the file tree is the information architecture. Everything else is derived.** If your directory structure makes sense to a new reader, the sidebar generates itself, URLs stay stable, and the search index is coherent. If it doesn't, no amount of sidebar config will rescue it â€” you'll just encode the confusion twice.

Three rules govern everything below:

1. **A broken link is a build failure, not a warning.** Docs rot silently. The only defense that works is a machine that refuses to publish rot. `onBrokenLinks: 'throw'` from commit one.
2. **Docs are code, reviewed like code.** They live in the repo, ship in the PR that changes the behavior, and get reviewed by the person who wrote the feature. A docs site fed by a separate backlog is wrong within a quarter.
3. **Every customization is a future upgrade tax.** Swizzling, custom plugins, and forked components carry a cost paid at every major version. Use CSS variables and theme config until they genuinely can't express what you need â€” which is later than you think.

If you cannot name the reader and the question they arrived with, you're writing a page nobody will find. That's an IA problem, and Docusaurus will happily render it beautifully.

## 2. Tech Stack

- **Docusaurus** â€” https://github.com/facebook/docusaurus â€” licensed **MIT**. React-based static site generator purpose-built for documentation: versioning, i18n, MDX content, and a docs-aware theme.
- **MDX v3** (with `@mdx-js/react`) â€” MIT. Docusaurus v3's content format. The v2 â†’ v3 jump is where most migration pain lives; see Â§3.3.
- **React 18** â€” MIT. Docusaurus v3 targets it; any custom component you write is a React component.
- **Algolia DocSearch** *(optional, free for qualifying open-source projects)* or **`@easyops-cn/docusaurus-search-local`** (MIT) for a self-hosted index.

This skill is an independent, original guide; it is not affiliated with or endorsed by the Docusaurus maintainers. All example code is original to this skill.

Recommended companions: `prettier` over `**/*.{md,mdx}` so prose diffs stay reviewable, `lychee` for external links, and a CI job that builds every PR â€” a docs site that only builds on `main` is a docs site that breaks on `main`.

## 3. Patterns

### 3.1 Sidebars: autogenerate the tree, curate the exceptions

Hand-maintained sidebars rot first. Every new page requires an edit in two places, and the one people forget is the sidebar. Let the filesystem drive it.

```js
// sidebars.js
module.exports = {
  docs: [
    // Curate the top: the first items are a guided path, not alphabetical.
    'intro',
    'install',
    { type: 'category', label: 'Quickstart',
      link: { type: 'doc', id: 'quickstart/index' },
      items: [{ type: 'autogenerated', dirName: 'quickstart' }] },
    // Let the tree drive the rest. These two lines cover 200 pages.
    { type: 'category', label: 'Guides', collapsed: false,
      items: [{ type: 'autogenerated', dirName: 'guides' }] },
    { type: 'category', label: 'Reference', collapsed: true,
      items: [{ type: 'autogenerated', dirName: 'reference' }] },
  ],
};
```

Ordering inside an autogenerated section comes from the file's frontmatter (`sidebar_position: 20`, `sidebar_label: Auth` for the short nav label, `title:` for the page and `<title>`). Per-directory, a `_category_.json` lets a folder name and place itself â€” `{ "label": "Guides", "position": 30, "link": { "type": "generated-index" } }` â€” so adding a folder adds a navigable section with no config edit at all.

The split that works: **explicit for the first ~5 items, autogenerated for the long tail.** New readers need a curated path through onboarding. Nobody needs a curated path through 80 reference pages â€” they arrive there from search. Number `sidebar_position` in gaps of 10, too: inserting between 2 and 3 means renumbering a file; inserting between 20 and 30 means typing `25`.

### 3.2 Broken links as a CI gate

The highest-value four lines in the entire config are these:

```js
// docusaurus.config.js
module.exports = {
  url: 'https://docs.example.com',
  onBrokenLinks: 'throw',
  onBrokenAnchors: 'throw',
  onBrokenMarkdownLinks: 'throw',
  onDuplicateRoutes: 'throw',
};
```

`throw` â€” not `warn`. A warning in a 900-line build log is a warning nobody reads. The gate that enforces it on every PR:

```yaml
# .github/workflows/docs.yml
on: [pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build          # fails the PR on any broken internal link
      - name: Check external links (non-blocking)
        continue-on-error: true
        run: npx lychee --exclude-path build/ 'docs/**/*.md*'
```

The distinction matters: **internal links block the PR; external links do not.** A third-party site being down at 2am should not block a teammate's typo fix. Run the external check nightly and file issues from it.

Docusaurus resolves internal links relative to the file, so link the file path â€” `[auth guide](./auth/tokens.md)` â€” not the URL. `/docs/auth/tokens` is a string nothing validates. The `.md` form is the only one the checker can verify, and it survives a `slug:` change.

### 3.3 MDX v3: the three things that will break your build

Docusaurus v3 uses MDX v3, which parses prose as JSX. This is the biggest source of "it worked in v2" pain, and it arrives in three flavors.

**Braces are JSX expressions.** `Set the template to {name}` tries to evaluate a variable called `name`, and the build dies. Fix with a code span (`` `{name}` ``) or an escape (`\{name\}`).

**`<` starts a JSX tag.** `Timeouts under <100ms are immediate` is a build error. Write `` `<100ms` `` or `&lt;100ms`. Generics bite hardest: an unquoted `Promise<Result>` in a sentence breaks the build. Anything code-shaped goes in a code span â€” a good habit regardless.

**HTML attributes are JSX attributes.** `class` becomes `className`, `for` becomes `htmlFor`, and every tag must close: `<br>` is an error, `<br />` is not.

The migration move that saves a week â€” make `.md` mean CommonMark again by setting `markdown: { format: 'detect' }` in `docusaurus.config.js`. From then on `.md` parses as CommonMark and only `.mdx` gets the JSX treatment, so prose stops exploding. Use this. Nine of ten docs pages are prose and want nothing to do with JSX parsing.

### 3.4 Versioning: the feature you should not turn on yet

`docusaurus docs:version 1.0` is one command and close to irreversible in practice. It snapshots your entire `docs/` tree into `versioned_docs/version-1.0/`, and from that moment every typo that exists in both versions is two edits, or three, or four.

**Do not version until all three are true:** (1) you have users on an old release who **cannot** upgrade, (2) the old docs are **materially different**, not just older, and (3) you've committed to fixing bugs in the old version. Until then, one `current` set plus a changelog is correct and free. Most projects that version early are documenting a version nobody runs. When you do version, cap the count:

```js
// docusaurus.config.js â€” presets â†’ docs
{
  lastVersion: 'current',
  versions: {
    current: { label: '2.x (latest)', path: '' },
    '1.0':   { label: '1.0 (maintenance)', path: '1.0', banner: 'unmaintained' },
  },
  onlyIncludeVersions: ['current', '1.0'],  // build time is linear in version count
}
```

`onlyIncludeVersions` is a real quality-of-life win in dev â€” a five-version site takes five times as long to build, and you're editing one of them.

### 3.5 Search: local first, DocSearch when you qualify

Two honest options and one trap:

- **`@easyops-cn/docusaurus-search-local`** â€” builds an index at build time, ships it to the client, no external service. Good to a few hundred pages. Works on private/internal docs. Costs a bundle download.
- **Algolia DocSearch** â€” free for qualifying open-source projects, crawler-driven, excellent ranking, no bundle cost. You apply, you wait, you tune a crawler config.

The trap is applying for DocSearch and shipping with no search for six weeks while you wait. Ship local search on day one; swap later â€” either way the config is five lines:

```js
themeConfig: {
  algolia: {
    appId: 'YOUR_APP_ID',
    apiKey: 'YOUR_SEARCH_ONLY_KEY',  // search-only. Never the admin key.
    indexName: 'your-index',
    contextualSearch: true,          // scopes results to current version + locale
  },
}
```

`contextualSearch: true` stops being optional once you version: without it a reader on 2.x gets 1.0 results and files a bug against behavior that hasn't existed for a year. And the key comment is not decoration â€” `docusaurus.config.js` is bundled to the browser, so the admin key in that object is a public write key.

### 3.6 Swizzling: wrap, don't eject

`swizzle` has two modes, and people click through the warning anyway. `npm run swizzle @docusaurus/theme-classic Footer -- --wrap` is safe: it renders the original component plus your addition. `... DocItem/Layout -- --eject` is not: it copies the source into your repo, where it is now yours, forever.

The discipline, in order: **`themeConfig` first** (navbar, footer links, colors, announcement bar are all config); **CSS variables second** (`src/css/custom.css` overriding `--ifm-color-primary` handles most branding without touching a component); **`--wrap` third**, because a wrapper survives upgrades â€” the internals stay upstream; and **`--eject` only when you've accepted the maintenance**, because an ejected `DocItem/Layout` makes every upgrade a manual three-way merge against a component you forgot you owned.

```jsx
// src/theme/Footer/index.js â€” the --wrap output. Original stays upstream.
import Footer from '@theme-original/Footer';
export default function FooterWrapper(props) {
  return (<><Footer {...props} /><StatusBanner /></>);
}
```

If a swizzle is marked "unsafe," that's the maintainers telling you the internals will change without a major version bump. Believe them.

### 3.7 Docs-as-code review flow and deploy

Docs build to a `build/` directory of static files. There is no server. Any static host works â€” S3 + CloudFront, GitHub Pages, Netlify, an nginx container. The deploy job is a mirror of the PR gate above: on push to `main`, `npm ci && npm run build` (where `onBrokenLinks: 'throw'` gates you a second time), then `aws s3 sync build/ s3://docs-example-com --delete`, then a CloudFront invalidation. Scope that workflow with a `paths:` filter on `docs/**`, `src/**`, and `docusaurus.config.js` â€” without it every unrelated push rebuilds and re-invalidates, and invalidations are the one line item here that actually costs money. Two review rules are worth enforcing in `CODEOWNERS` on top of that:

- **The PR that changes behavior changes the docs.** Not a follow-up issue. A follow-up issue is a promise, and docs are where promises expire.
- **Deploy previews on every PR.** Reviewing rendered docs in a diff is reviewing MDX, not prose. Half the errors â€” a broken table, an unparsed admonition, a wrong fence language â€” are invisible in the diff and obvious in the preview.

## 4. Anti-patterns

- **`onBrokenLinks: 'warn'`.** You configured your build to tell you about the rot and then ship it anyway. Nobody reads a warning in a green build. Set `throw` and let the machine hold the line.
- **A hand-maintained 400-line `sidebars.js`.** Every new page is an edit in two files, and the sidebar is the one people forget. Autogenerate the long tail; curate only the guided path at the top.
- **Versioning in month two.** You snapshotted docs for a release nobody runs, and every typo fix is now N edits. Version when you have users who cannot upgrade â€” not before.
- **`--eject` because the prompt offered it.** You forked an internal component and signed up for a manual merge on every upgrade. Try `themeConfig`, then CSS variables, then `--wrap`. Eject last, deliberately.
- **Unquoted `<` and `{` in MDX prose.** `Promise<Result>` and `{count}` in a sentence are JSX, and MDX v3 fails the build. Code spans around anything code-shaped.
- **Absolute URL links between docs pages.** `/docs/auth/tokens` is a string nothing validates, and it dies silently the moment a `slug` changes. Link `./auth/tokens.md` so the checker sees it.
- **A separate docs repo with its own backlog.** Docs in another repo ship on another schedule, which means they ship wrong. Same repo, same PR, same reviewer as the behavior they describe.
- **DocSearch's admin API key in `docusaurus.config.js`.** That file is bundled to the browser. Only the search-only key belongs there â€” anything else is a write key in public source.
- **Shipping with no search while the DocSearch application is pending.** Six weeks of readers who can't find anything. Ship local search now; swap five lines later.
- **Reviewing docs PRs from the diff alone.** Broken tables and unparsed admonitions are invisible in raw MDX and obvious in a deploy preview.

## 6. Example Output

Prompt given with this skill loaded: *"Add a Reference section. It'll grow to ~60 pages. Some pages need a 'requires v2.3+' badge in the intro."* Expected shape of the answer:

```js
// sidebars.js â€” one entry for 60 pages, and it never needs editing again
{
  type: 'category', label: 'Reference', collapsed: true,
  link: { type: 'generated-index', title: 'API Reference',
          description: 'Every public method, alphabetical.' },
  items: [{ type: 'autogenerated', dirName: 'reference' }],
}
```

Alongside it, `docs/reference/_category_.json` holding `{ "label": "Reference", "position": 40 }` so the folder names and places itself. Then `docs/reference/create-session.mdx` â€” frontmatter (`sidebar_position: 20`, `sidebar_label: createSession`, `title: createSession()`), an `import SinceBadge from '@site/src/components/SinceBadge';`, and a body that renders `<SinceBadge version="2.3" />` under the H1. Every code-shaped token sits in a code span, so an unquoted `Promise<Session>` never reaches the JSX parser, and the "see also" link is written `[session lifecycle](./lifecycle.md)` so `onBrokenLinks: 'throw'` verifies it at build time.

```jsx
// src/components/SinceBadge.jsx â€” a plain component. No swizzle needed.
export default function SinceBadge({ version }) {
  return (
    <span className="badge badge--secondary"
          aria-label={`Requires version ${version} or later`}>requires v{version}+</span>
  );
}
```

Note what the output does *not* contain: no swizzled `DocItem`, no per-page sidebar entry, no versioned snapshot. The badge is an ordinary MDX import â€” the lightest possible customization â€” and the 60-page section is one autogenerated line the filesystem maintains for you.
